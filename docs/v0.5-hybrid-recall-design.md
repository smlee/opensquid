## Locked decisions (one line each)

| # | Decision | Why (1 sentence) |
|---|---|---|
| 1 | Promote serve.rs `score()` to a shared `engine::scoring::score_text_match(query, description, body) -> f32` module. | Prevents drift between lesson + memory text-paths. |
| 2 | Extend `memory.search` RPC with `mode: "semantic" \| "text" \| "hybrid"` (default `"semantic"` for back-compat). | Single endpoint, additive, opensquid stays simple. |
| 3 | Description weighting: `(score(q, desc) * 2.0 + score(q, body)) / 3.0`, capped at 1.0. | 2x mirrors industry default (Tantivy/ES BM25); average prevents desc drowning body. |
| 4 | Single `min_similarity` threshold (0.5 default) applied uniformly to RAW per-source scores BEFORE the RRF merge — same scale for semantic and text since both are 0.0–1.0 normalized. (**Refined during build:** original wording said "post-merge"; RRF scores live in `[0, ~0.033]` and can't share the threshold, so the filter was moved to engine-side pre-RRF.) | Defer dual-threshold complexity until empirical ROC data justifies it. |
| 5 | RRF accumulator code (`recall.ts:82–120`) needs NO change — the `existing.rrf_score += score` path already handles same-id-twice; just gets exercised for the first time. | Existing code is correct; activation is the deliverable. |
| 6 | Performance: scan all memory frontmatter on text-search calls. Defer in-memory description cache to v0.6. | Acceptable up to ~1k memories (the realistic 2026 corpus); cache is premature optimization. |
| 7 | Hand-roll the score function (Option G.4); reject Tantivy / bm25 crate / Meilisearch for v0.5. | Existing `score()` already proven for lessons; new dep = bloat per user feedback. |
| 8 | Keep substring bonus (+0.3) in text-match; rely on RRF to fuse it with semantic. | The bonus mechanism works; the gap was only that no text path existed for memories. |
| 9 | Reject simpler "post-merge filter + description-substring override" alternative. | Special-case smell; RRF solves it cleanly without a sentinel path. |
| 10 | Backward-compat: all new params + result fields are additive. v0.4 callers unaffected. | The wedge of "ship dark, light up later" pattern that worked in v0.4 Phase 5a. |

## Implementation order (4 commits)

### Commit 1 — Engine: `engine::scoring` module
**Files:** new `src/engine/scoring.rs`; `src/engine/mod.rs` re-export; `src/serve.rs` lesson_recall now imports from `crate::engine::scoring`.
- `pub fn score_text_match(query: &str, description: &str, body: &str) -> f32`
- Internal helpers: `tokenize` (whitespace + lowercase), `score_token_overlap`, `substring_bonus`
- Tests: 6 cases (empty query, exact match, partial token match, substring-only, description-only, body-only)
- Promote, don't duplicate. lesson_recall's existing inline `score()` is deleted.

### Commit 2 — Engine: `memory.text_search` + `memory.search` mode parameter
**Files:** `src/engine/memory/store.rs` (new `text_search()` fn); `src/serve.rs` (extend `MemorySearchParams`, dispatch `mode`).
- `pub async fn text_search(ctx, storage, query, k, scope_filter) -> Result<Vec<MemoryRef>, EngineError>` — scans `memories_prefix`, scores via `score_text_match`, returns top-k by score, soft-fails on parse errors (mirrors `search()` resilience).
- `MemorySearchParams.mode: Option<SearchMode>` (default `Semantic`); `enum SearchMode { Semantic, Text, Hybrid }` with `#[serde(rename_all = "snake_case")]`.
- Dispatch: `mode = Semantic` → existing path. `mode = Text` → new `text_search`. `mode = Hybrid` → RRF in serve.rs combining both.
- `MemoryRef` gets a new optional `source: Option<HitSource>` enum (`{Semantic, Text, Both}`) — additive.
- Tests: 4 cases per mode (empty store, semantic-only-hit, text-only-hit, both-sources).

### Commit 3 — opensquid: pass `mode: "hybrid"` from `recall` handler
**Files:** `src/engine-client.ts` (`searchMemory` accepts new `mode` param); `src/index.ts` (recall handler defaults to `mode: "hybrid"`).
- Engine-client: type stub for `SearchMode` matching engine wire; pass-through.
- Recall handler: when fanning out to `engine.searchMemory`, set `mode: "hybrid"` unless caller explicitly passes another value.
- RRF merge in `recall.ts` is unchanged — it's now exercised on same-id collisions for the first time, which the existing accumulator code handles correctly.
- Tests: re-run the dogfood query "Gianna" — must return mem-a0cdce30 with rrf_score above threshold.

### Commit 4 — Tests + CHANGELOG + verification
- End-to-end test script (mirrors `/tmp/v04-rrf-test.mjs`) covering: name lookup ("Gianna"), descriptive query ("how do we deploy"), out-of-domain ("bakery"), substring-only ("Heroku" against alpha-project memory).
- Engine CHANGELOG (Unreleased section): mention scoring module + memory.text_search + mode param.
- opensquid CHANGELOG: hybrid recall, dogfood-driven calibration of v0.4 threshold (0.5 stays the default; finally has a real signal to clear it).

## Out of scope for v0.5

- BM25 / Tantivy migration (defer to v0.6 if scale demands it)
- In-memory description cache (defer to v0.6 — premature for current corpus size)
- Fuzzy / typo / stemming (defer indefinitely; let the LLM rephrase)
- FTS5 SQLite index (architectural shift, not a v0.5 fit)
- Per-mode dual thresholds (defer until ROC curves on semantic vs text show divergent distributions)
