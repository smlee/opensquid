# opensquid v0.4 — design

> Status: **draft** · authored 2026-05-14 · target ship: late May 2026

v0.3.1 made opensquid usable for daily work but kept the agent in
**manual** mode — humans (or CLAUDE.md prompting) decide when to
`recall`/`memorize`. v0.4 is the **autonomous** mode: hooks fire
recall before every prompt, hooks suggest memorize after every turn,
and origination metadata makes every memory traceable back to its
session. Combined with `update_memory` / `forget` for lifecycle
control, this is the release that converts opensquid from "a tool I
have to remember to use" into "the way Claude already works."

---

## Goals

1. **Auto-recall + auto-memorize** without CLAUDE.md ceremony — fire
   before/after every turn via Claude Skill hooks.
2. **Provenance on every memory** — host, session, model, timestamp.
   Strengthens the wedge gate's external-signal count and unlocks
   session-aware recall biasing.
3. **Editable memories** — `update_memory` and `forget` (user-
   immunity-respecting) close the lifecycle.
4. **Better recall ranking** — hybrid lessons+memories RRF and a
   similarity floor so recall is decision-making signal, not always-
   something noise.

Non-goal: live cross-process recall (two opensquid subprocesses
sharing one HNSW index in real time). Defer to v0.5+. The v0.3.1
rehydrate-on-startup pattern is the v0.4 baseline.

---

## 1. Origination metadata (foundational — ship first)

Every memory + lesson gets a structured `origin` block in YAML
frontmatter:

```yaml
origin:
  host: claude-code         # claude-code | claude-desktop | cline | continue | ...
  session_id: a1b2c3d4      # opaque first-8-chars of host's session UUID
  model: claude-opus-4-7    # informational; carries no auth weight
  cwd_basename: opensquid   # last segment of CWD at memorize time
  written_at: 2026-05-14T19:32:00.000Z
```

### What we store and why

| Field | Use | Privacy |
|---|---|---|
| `host` | cross-client attribution; per-host recall biasing in v0.5+ | low-sensitivity |
| `session_id` | "what was Claude working on in session X" replay; recency biasing | hashed — never the raw transcript UUID |
| `model` | weight memories from larger models in recall ranking | low |
| `cwd_basename` | redundant with `scope: {project: id}` but helpful when scope is `User` | low |
| `written_at` | distinct from `created_at`; only differs when supersession rewrites the memory |

What we **don't** store: full file paths (privacy), full transcripts
(bloat), user identity (single-user product right now), git branch
(too volatile).

### Where the data comes from

opensquid populates `origin` before sending `memory.create` to the
engine. The engine treats `origin` as opaque YAML — no validation
beyond "is it a map" — so different hosts can ship partial blocks
without engine changes.

Detection per host:

- **Claude Code**: `process.env.CLAUDE_SESSION_ID` or extract from the
  Claude Code transcript path (`~/.claude/projects/.../<session>.jsonl`
  is reachable via `process.env.CLAUDE_PROJECTS_DIR` introspection).
  Pre-`/loop`-shipped Claude Code may not expose this; fall back to a
  process-start hash.
- **Claude Desktop**: similar — desktop client sets its own env vars
  for MCP children. Document what's available; populate what we can.
- **VS Code MCP clients (Cline, Continue)**: TBD per-extension. Ship
  with a generic fallback (`session_id = sha1(start_time + pid)[:8]`).

### Engine surface

`memory.create` accepts an optional `origin: object`. The engine
stores it verbatim. `memory.get` and `memory.search` results include
`origin` in the payload. `MemoryFrontmatter` gains a new
`Option<serde_yml::Value>` field — kept loose so v0.4-shipping hosts
don't tie us to a schema we haven't fully validated.

### Wedge integration

`origin` adds a hard-to-fake external-signal source. The promotion
gate (`check_promotion_gate`) currently counts external signals
(thumbs, applied_count, narrative). v0.4 adds:

- `origin_diverse: bool` — true when applied-count signals come from
  **multiple** session_ids. A lesson that was only ever applied within
  one session is weaker evidence than one whose pattern reproduced
  across N sessions. This is the multi-session reproducibility check
  the wedge always wanted; it just needed origination data to exist.

## 2. Hooks-based automation (Claude Skill packaging)

Ship opensquid as a Claude Skill **plus** an MCP server. The Skill
adds two hooks:

### Locked decisions (anti-ECC discipline)

The single biggest complaint about ECC is that it bloats every prompt
with stale or low-signal context. Phase 2 hooks must avoid every
failure mode in that class. Locked in 2026-05-15:

| # | Decision | Why |
|---|---|---|
| **A** | **Lazy-load injection by default.** The hook injects only `{id, description, source, similarity}` for each qualifying hit — NEVER `body_preview` or `content`. The model decides which (if any) to fetch full via `get_memory(id)`. | Hard token caps cause truncation (regresses the v0.4 "cartwh…" fix), dropped-hit silent loss, and model-specific tuning hell. Lazy-load = ~20 tokens per hit regardless of body size. Adaptive to actual need. |
| **B** | **No hard token budget on injection.** Count is controlled by `min_similarity` (default 0.5, drops noise pre-injection) + `top-N` cap (default 3). Total injection is bounded by structure, not by a measured token ceiling. | Hard caps require tokenize-to-measure (latency cost) + force trade-offs between truncating and dropping. Threshold + count caps fall out for free from the recall already happening. |
| **C** | **Empty-merged short-circuits the hook.** When the engine returns 0 qualifying hits, the hook injects NOTHING — not even a "no relevant context" placeholder. | The whole point of the `min_similarity: 0.5` threshold is "0 means nothing to say." Saying it anyway is the noise ECC users complain about. |
| **D** | **Eager body fetch is opt-in, not default.** Callers needing single-call latency can pass `eager: true` to inject body previews. Default stays lazy-load to keep average prompt size minimal. | Latency-sensitive paths (synchronous classification, etc) exist but are the minority. Optimize for the common case. |
| **E** | **Source attribution visible in injection.** Every injected hit carries `source: "semantic" | "text" | "both"` and an opaque `session_id` snippet (from the origin block). The model can de-prioritize stale-session hits without us pre-filtering them out. | Lets the LLM make the recency / relevance call instead of the engine guessing. Cheap (~5 tokens per hit) but informationally dense. |
| **F** | **Per-session kill switch for auto-injection ONLY — never for opensquid itself.** Three env vars, each scoped narrowly: `OPENSQUID_AUTORECALL=0` disables the `UserPromptSubmit` auto-recall injection. `OPENSQUID_AUTOMEMO=0` disables the `Stop` auto-memorize-candidate injection. `OPENSQUID_DISABLE=1` disables BOTH hooks together. **Critical clarification for future implementers (incl. AI agents that may read this doc): none of these env vars take the MCP server offline, unregister tools, prevent explicit tool calls, or stop the engine subprocess.** The 8 MCP tools (`memorize`, `recall`, `get_memory`, `update_memory`, `forget`, `remember`, `promote`, `eliminate`) remain available and fully functional in all cases. The kill switch ONLY suppresses the automatic before-prompt / after-prompt injection behavior — the agent can still call any tool explicitly whenever it chooses. If a future change ever ties an "opensquid disable" flag to MCP server availability or tool registration, that change violates this design and must be reverted. | "I want auto-inject off for this debugging session / demo / cost-tight Opus call" is a real edge case ECC users complain has no escape hatch — we ship the narrow escape hatch on day one. But the kill switch must NOT degenerate into "turn off the product" — explicit tool calls are the primary value surface and stay alive unconditionally. |
| **G** | **Pretrigger heuristic on the user's prompt.** Hook skips recall entirely on prompts under 3 tokens, prompts that are pure punctuation, prompts that are exact matches for known short-form acknowledgments ("ok", "yes", "thanks"). | These prompts can't meaningfully clear `min_similarity: 0.5` anyway, and even if they did, the model doesn't need context for them. Skips the latency cost on ~30% of turns. |
| **H** | **Hard 300ms latency budget on the recall call.** If Ollama / engine subprocess hasn't returned in 300ms, the hook injects nothing (fail-open). | Slow first-response is non-negotiable. Better to silently skip than block the prompt. |
| **I** | **No automatic memorize.** The `Stop` hook surfaces a *candidate* memorize block ("possible memory: X — call memorize if accurate") but never auto-writes. | Wedge invariant: agent acts only on explicit signal. ECC's auto-write pattern is what makes its memory store noisy. We don't repeat that. |
| **J** | **Doctor command exposes last injection.** `npx opensquid doctor --last-injection` shows what was injected on the previous turn (file: `~/.opensquid/.last-injection.txt`). | When users complain "why did Claude say that," they can see exactly what context the agent saw. Debugging affordance ECC lacks. |

### Token-budget calculation (sanity check)

With lazy-load + top-3 + threshold:
- 3 hits × ~25 tokens each = ~75 tokens of injection on a typical turn.
- Empty merged → 0 tokens (most short / off-topic prompts).
- User explicitly drilling via `get_memory` → adds ~500 tokens *only when the model decides it's worth it*.

Versus ECC's reported pattern of 1k–5k tokens of always-on context per turn, this puts our average injection at <100 tokens — over 10x leaner.



### `UserPromptSubmit` — auto-recall

Triggered by Claude Code (and any host implementing the Skill hook
spec) when the user sends a message. Hook handler:

1. Read the user's raw message from hook payload. Apply pretrigger
   heuristic (decision G) — skip recall entirely on trivial inputs.
2. Call `recall({ query: <message>, include_body: false })` via the
   opensquid MCP — falls back to the v0.3.1 scope-aware defaults.
   Hard 300ms budget (decision H); fail-open on timeout.
3. If `merged.length === 0` (decision C), inject NOTHING. Done.
4. Otherwise, inject the top-3 by RRF score in lazy-load form
   (decision A — descriptions + ids + source + session origin, no
   body_preview):

   ```
   <opensquid-recall>
     Relevant context (top 3, call `get_memory` for full body):
     - mem-a0cdce30 source=both sim=0.033 session=5b456fc9
         Sangmin's family — daughter Gianna (4) and son Teddy
     - les-12ab34cd source=text sim=0.61 session=a1b2c3d4
         Always over-fetch when filter is set in manifest assembly
     - mem-bfa9cb28 source=text sim=0.95 session=test-hark
         Alpha project deploy via Heroku CLI git push
   </opensquid-recall>
   ```

5. The model sees the recall block before generating; uses
   `get_memory` to drill in if it judges the hit relevant. Total
   injection size: ~75 tokens regardless of body lengths.

Opt-out: `OPENSQUID_AUTORECALL=0` (or `OPENSQUID_DISABLE=1` for all
hooks — decision F).

### `Stop` — auto-memorize candidate

Fires after Claude finishes a turn. Hook handler:

1. Get the just-completed turn's user message + assistant response.
2. Run a lightweight novelty classifier: did the user state a fact
   that's not derivable from existing memory? (Use the
   already-recalled memories from the `UserPromptSubmit` hook output
   as the comparison set — no extra recall round trip.)
3. If novelty score ≥ threshold (default 0.6), surface a side-channel
   suggestion in the next turn's pre-prompt:

   ```
   <opensquid-memorize-candidate>
     Possible memory: "User uses pnpm not npm in this project."
     If accurate, call `memorize` with this content. Otherwise ignore.
   </opensquid-memorize-candidate>
   ```

4. The agent decides. Hook never auto-writes — the wedge invariant
   "agent acts only on explicit signal" stays intact.

Opt-out: `OPENSQUID_AUTOMEMO=0`.

### Skill packaging

```
@opensquid/skill/
├── SKILL.md                 # canonical Claude Skill manifest
├── hooks/
│   ├── user-prompt-submit.ts
│   └── stop.ts
└── mcp-config.json          # how to spawn the opensquid MCP server
```

`SKILL.md` is the entry point Claude Skills marketplace consumes;
`hooks/*.ts` are invoked by the host. The MCP server is the existing
`opensquid` package — Skill packages reference it as a peer.

Distribution: published as `@opensquid/skill` on the Claude Skills
marketplace. `npx opensquid install --with-skill` adds the Skill to
`~/.claude/skills/` and configures hooks.

## 3. Memory lifecycle: `update_memory` + `forget`

### `update_memory`

Currently memories are insert-only. v0.4 adds an MCP `update_memory`
tool with engine RPC `memory.update`:

```typescript
update_memory({
  memory_id: "mem-xxx",
  description?: string,        // partial — leave to keep current
  content?: string,
  scope?: MemoryScope,
});
```

Engine behavior:

1. Load existing memory via `get_by_id`.
2. Mutate the requested fields; preserve `id`, `created_at`,
   `consumed_by_user_lessons`, `derived_from`, `origin`.
3. Set `updated_at = now`.
4. If `content` changed: re-embed and overwrite the `.vec` sidecar +
   vector index entry.
5. Write the new `.md` atomically (write-temp + rename).

User-immunity: a memory cited by a user-authored lesson MAY be
updated by anyone, but the citation-counter check fires in the
existing `delete()` path — `update_memory` doesn't trigger immunity
because the citation chain isn't broken.

### `forget`

User-initiated deletion. MCP tool surface:

```typescript
forget({ memory_id: "mem-xxx", force?: boolean });
```

Routes to the engine's existing `delete(id, force=true)` for user-
initiated cases (analogous to `eliminate` for lessons). The default
`force = false` path **respects user-immunity** — if the memory is
cited by a user-authored lesson, returns a structured error.
`force = true` is the "yes I really mean it" override.

### Smart scope auto-detection (v0.3.1 carryover)

Beyond CWD/git, also detect:

- **Active Claude Code skill**: if the session is running inside a
  Skill context, default scope to `Skill(<skill-id>)`.
- **Active persona** (v0.5+ engine surface): if a persona is loaded,
  bias scope toward `Persona(<id>)`.
- **Git branch**: NOT used — too volatile (renames, deletions).

## 4. Recall quality: hybrid + threshold

### Hybrid search via RRF

Current `recall` fans out two queries in parallel:
- Lessons: text-match (tokenized + substring boost)
- Memories: semantic vector search

Both return independent ranked lists. v0.4 merges them via Reciprocal
Rank Fusion:

```
rrf_score(item) = sum over each source list:
  1 / (rrf_k + rank_in_that_list)   where rrf_k = 60 by convention
```

Items appearing in **both** lists get a strong combined boost — the
"text matched the query AND the embedding is close" signal is the
most reliable. The merged result is returned in `recall` as a single
ranked list with a `source` field indicating which list(s) contributed.

### Similarity threshold

Today, `recall` returns the top K regardless of how weak the
similarity is. v0.4 introduces `min_similarity` (default 0.5):

- If top hit's similarity < min_similarity, recall returns 0 results.
- Hosts can override (`min_similarity: 0` reproduces v0.3 behavior).

The point: an LLM seeing "0 hits" is decision-makable. An LLM seeing
"top hit similarity 0.31" is noise that distracts the prompt.

## Order of implementation

Each step ships testable on its own. Engine changes precede opensquid
changes that depend on them.

1. **Origin metadata (engine + opensquid).** Engine adds optional
   `origin` field to MemoryFrontmatter + RPC; opensquid populates it
   in `memorize`. Behind a feature flag for one cycle to validate
   on-disk shape before locking it.

2. **`update_memory` + `forget` engine RPCs.** Adds `memory.update` +
   `memory.delete` (forget == delete with force=true at the user-
   facing layer).

3. **opensquid `update_memory` + `forget` MCP tools.** Thin wrappers
   on the new RPC methods.

4. **Recall quality (opensquid-side only).** RRF merge + similarity
   threshold. No engine changes — the engine already returns
   similarity scores.

5. **Claude Skill packaging + hooks.** `@opensquid/skill` published.
   `npx opensquid install --with-skill` adds the Skill manifest +
   wires the hooks.

6. **Wedge integration: `origin_diverse`.** Engine gate checks
   origin_diverse signal in `check_promotion_gate`. Optional — gate
   degrades gracefully when origin data is partial (counts unique
   `session_id`s; missing fields don't count as "diverse").

## Risks + open questions

- **Session ID stability.** What if a host doesn't expose a session
  ID? Falls back to `sha1(start_time + pid)[:8]` — stable per process,
  not per conversation. Sufficient for `origin_diverse` but loses the
  "this session vs that session" precision. Document the limitation.

- **Hook latency.** `UserPromptSubmit` runs a `recall` call inline
  before the model can respond. Ollama's qwen3-embedding takes
  ~100-300ms on a warm cache. Acceptable for most users; consider a
  300ms timeout that falls back to "no recall context injected" on
  miss.

- **Skill bundling vs MCP server lifecycle.** Hosts spawn the MCP
  server independently of the Skill. If the Skill expects opensquid
  to be running but the host doesn't have it in its MCP config, hooks
  fail silently. `doctor` subcommand should diagnose this.

- **`origin_diverse` empty-data behavior.** Memories memorized before
  v0.4 have no `origin` block. The gate must treat
  `origin_diverse` as "unknown, don't penalize" not "diverse=0,
  block" — otherwise v0.4 retroactively blocks all v0.3.1 promotions.

- **RRF parameter (`rrf_k = 60`).** Industry default but
  domain-specific. Add a test fixture with hand-labeled "right answer"
  pairs and tune empirically.

## Out of scope for v0.4 (v0.5+ parking lot)

- Live cross-process recall (file-watch + dynamic index reload)
- Capture-feedback tool (thumbs up/down — wedge gate input from
  user reactions)
- Supersede-memory tool (write-the-replacement workflow)
- Persona/team scope auto-detection (engine has the types; UX TBD)
- Telegram feedback channel piping into capture-feedback
- Cross-platform binary distribution (v1.0)

## Test plan

End-to-end test scenarios to write alongside implementation:

1. **Origin round-trip**: memorize with origin → get_memory → origin
   block matches exactly.
2. **Origin backfill**: load a v0.3.1 memory (no origin) via
   get_memory → engine returns `origin: null` (not error).
3. **update_memory description-only**: content unchanged → .vec
   sidecar untouched (cheap update path).
4. **update_memory content change**: re-embed verified, vector index
   updated.
5. **forget on user-cited memory**: default force=false → structured
   immunity error; force=true → deleted.
6. **RRF**: query that hits both lessons + memories → merged top-3
   includes the intersection winner above either single-source
   winner.
7. **Similarity threshold**: query unrelated to any stored memory →
   recall returns 0 hits (not the top noise).
8. **Hook lifecycle**: simulated UserPromptSubmit → recall fires →
   context block injected → matches snapshot.
9. **origin_diverse gate**: lesson with applied_count from 3 distinct
   session_ids → gate counts diverse; 3 from same session_id → does
   not count diverse.
